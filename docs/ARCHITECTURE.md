# Algorithmia Architecture

## System Overview

Algorithmia currently has two runtime surfaces:

1. The browser game under `web/` is the player-facing product.
2. The `algorithimia/` Python package is a legacy deterministic encounter, validation, and trace reference.

The browser game is a hybrid application. Phaser handles real-time combat simulation, while ordinary DOM modules handle text-heavy RPG and programming interfaces. This keeps movement and collision inside a game engine without forcing the code editor, dialogue, and menus into canvas-only UI.

## Browser Startup

```text
web/index.html
  -> vendor/phaser.min.js
  -> web/js/main.js
       -> title.js
       -> room.js / chapter*.js
       -> encounter modules
```

`vendor/phaser.min.js` is generated by `npm run build` from the pinned npm dependency. It is not source code and is not committed.

## Browser Ownership Boundaries

| Area | Owner | Responsibility |
| --- | --- | --- |
| Screen routing | `main.js`, `title.js` | Title, campaign, Arcade, chapter transitions |
| Save data | `state.js` | Defaults, `localStorage`, reset and persistence |
| Shared vitals | `combatState.js`, `battleHud.js` | HP, Focus, shared combat readouts |
| Queueworks map | `room.js` | Chapter 0 movement, interactions, campaign win callback |
| Later maps | `chapter1.js`-`chapter4.js` | Room graphs, enemies, puzzles, secrets, chapter progression |
| LDtk import test | `chapter5.js`, `web/data/ldtk/` | Runtime fetch/render of LDtk JSON, collision grid, doors, items, buttons, and secret walls |
| Real-time arena | `slimeArenaEngine.js` | Phaser scene, player movement, collisions, waves, phases |
| Slime encounter coordination | `slimeArenaBattle.js` | DOM commands, Repair overlay, sound, save state, arena callbacks |
| Slime balance | `slimeArenaConfig.js` | Arena dimensions, HP, timing, phase names, speed |
| Slime hazard patterns | `slimeArenaPatterns.js` | Column, merge, and spiral hazard placement |
| Slime repair content | `slimeRepairTasks.js` | Prompts, starters, examples, hints, and cases |
| Workshop editor | `workshopData.js`, `workshopValidation.js`, `workshopEditor.js` | Visual room drafting, level-pack JSON, reachability checks, local browser drafts |
| Browser Python subset | `pythonRepairRuntime.js` | Worker execution, timeout, metrics, limited translation |
| Earlier battles | `battle.js`, `ticketBattle.js`, `codeBattle.js`, `bogoBossBattle.js` | Pre-Phaser encounter implementations |
| Dialogue | `dialogue.js` | Queueing, typewriter text, completion callbacks |
| Assets and sprite helpers | `playerSprite.js`, `sprites.js`, `pixelart.js` | PNG placement and matrix placeholder rendering |

Do not put DOM mutation into Phaser pattern code. Do not put collision or per-frame movement into a DOM encounter controller. Cross that boundary through callbacks.

## Sorting Slime Runtime Flow

```text
Campaign collision or Arcade selection
  -> startSortingSlimeArenaBattle()
  -> startSlimeArena()
  -> continuous Phaser hazard pattern
  -> player reaches Sorting Slime
  -> Attack / Use / Repair / Guard
       Repair -> Web Worker -> public and sealed cases
              -> shield breach -> changed arena behavior
  -> boss HP reaches zero
  -> caller-provided win callback
       campaign: persist gate progress
       arcade: return to encounter selection only
```

The caller owns rewards. The arena owns combat. This prevents Arcade battles from accidentally clearing campaign gates.

## Adding a Phaser Encounter

Prefer three layers:

1. `*ArenaEngine.js`: deterministic combat state and Phaser objects.
2. `*Battle.js`: DOM controls, editor, audio, vitals, and game-state coordination.
3. Campaign/Arcade entry points: dialogue, reward, and return destination.

An encounter engine should expose narrow commands such as `attack`, `guard`, `openRepair`, `resumeRepair`, `retry`, and `stop`. The controller should not reach into scene internals.

Boss patterns need:

- a readable telegraph;
- a survivable route or response;
- deterministic timing boundaries that tests can observe;
- explicit cleanup of timers, tweens, input handlers, and entities;
- reduced-motion behavior where visual motion is nonessential;
- a defeat path and admin completion path.

## Browser State

`state.js` stores a single versioned `localStorage` object. Every persisted field needs a default so older saves can merge forward safely.

Current categories include:

- player HP and Focus;
- defeated enemies and bosses;
- room puzzle state;
- discovered secrets;
- Graphreach diamonds;
- options and admin-adjacent state.

Do not persist temporary Phaser objects, timers, DOM nodes, or repair workers.

Workshop Mode uses its own `localStorage` key for editor drafts. A live Vercel deployment can save, load, import, and export level JSON in the browser, but it cannot write official campaign files back to GitHub without a future authenticated backend. Official campaign content should still be reviewed and committed as repository data.

## Browser Python Execution

There are currently two browser code paths:

- `pythonRepairRuntime.js` and similar older workers translate a limited Python-shaped subset into JavaScript inside a disposable Web Worker.
- The Python package runs actual Python in a subprocess for local CLI validation.

The browser subset supports the current sorting lesson but is not full Python. It must not be presented as a security sandbox or general CPython environment. Pyodide or another real browser Python runtime is expected before encounters depend on broader language features.

Every browser execution path must retain:

- main-thread isolation;
- a hard timeout;
- bounded input and output;
- public and sealed cases;
- non-disclosing sealed failures.

See `SAFE_EXECUTION.md` for the Python package's separate security notes.

## Legacy Python Package

The package name `algorithimia` is historical and intentionally preserved for import and CLI compatibility. New project-facing text must use **Algorithmia**.

Key modules:

- `algorithimia.cli`: terminal entry point.
- `algorithimia.encounters`: encounter data and fixtures.
- `algorithimia.engine`: validation flow.
- `algorithimia.language.python_adapter`: constrained subprocess execution.
- `algorithimia.visualizers`: structured deterministic trace events.
- `algorithimia.trace_viewer`: static trace HTML export.
- `algorithimia.game_shell`: historical generated browser shell, not the current `web/` game.

The legacy package remains useful as an oracle for deterministic encounter rules. New player-facing work belongs in `web/` unless explicitly extending the reference engine.

## Assets

Runtime assets belong under `web/assets`:

- `characters/<name>/`: directional exports or project-local sprites;
- `audio/`: sound effects and later music;
- generated Phaser runtime: `web/vendor/`, never committed.

Character metadata may preserve original generator descriptions. Runtime code should reference stable project-relative paths rather than top-level raw export folders.

LDtk test maps live under `web/data/ldtk/` so Vercel can publish them with the static site. The current Chapter 5 importer reads LDtk JSON directly, uses the `Collisions` IntGrid for blocking, renders LDtk entities with Algorithmia placeholder sprites, and supports simple items, keyed doors, buttons, and secret walls. It does not yet use external LDtk tileset images or convert LDtk maps into official campaign data files.

## Tests

The test layers are complementary:

- Python unit tests validate encounter contracts, adapters, traces, and historical shell output.
- `test_web_room_maps.py` validates dimensions and reachability for required room interactions.
- JavaScript syntax checks catch module parse failures.
- `slime_arena_browser_smoke.cjs` verifies direct, campaign, and Arcade launch paths plus keyboard commands, Repair input, Python results, assets, victory, and mobile canvas fitting.
- `workshop_browser_smoke.cjs` verifies the Workshop can open directly, paint tiles, place entities, edit dialogue and encounter data, save a draft, reload it, and export JSON.
- `ldtk_chapter_browser_smoke.cjs` verifies the LDtk JSON publishes, Chapter 5 renders, and an LDtk item can be collected from the imported map.

Passing unit tests does not prove a canvas rendered correctly. Passing a screenshot does not prove a room route is reachable. Run both relevant layers.

## Deployment

Vercel follows `vercel.json`:

1. `npm install`
2. `npm run build`
3. publish `web/`

Branch pushes receive Preview deployments. `main` is production. The build must regenerate `web/vendor/phaser.min.js`; a missing vendor file in Git is expected.

## Known Technical Debt

- Only Sorting Slime uses the Phaser arena.
- Later battle modules duplicate Python-subset worker logic.
- Chapter files own large inline room definitions and will eventually benefit from data extraction.
- Workshop exports are not yet consumed by campaign rooms directly.
- Chapter 5 is a proof-of-import LDtk route, not a final authored campaign chapter.
- `style.css` is monolithic and should be split only with visual regression coverage.
- The Python package name is misspelled for historical compatibility.
- Mobile battle input is keyboard-first; dedicated touch combat controls are not complete.

Treat these as explicit migration work, not reasons for opportunistic rewrites inside unrelated encounters.
