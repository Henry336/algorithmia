# Algorithmia Architecture

## Runtime Surfaces

Algorithmia has two runtime surfaces:

1. `web/` is the player-facing browser game and the production Vercel output.
2. `algorithimia/` is the older deterministic Python validation and trace package. Its misspelled package name remains for compatibility.

The browser game is hybrid by design:

- Phaser renders continuous exploration and real-time boss arenas.
- DOM screens render menus, dialogue, coding editors, test results, options, Arcade, and Workshop.
- Small callbacks cross that boundary. Phaser engines do not directly edit coding panels, and DOM controllers do not run collision every frame.

## Startup

```text
web/index.html
  -> web/vendor/phaser.min.js
  -> web/js/main.js
       -> title.js
       -> campaignAtlas.js
            -> campaignAtlasData.js
            -> campaignAtlasArt.js
            -> campaignAtlasEncounters.js
       -> encounter controllers
       -> workshopEditor.js
```

`web/vendor/phaser.min.js` is generated by `npm run build` from Phaser 3.90. Do not edit or commit it.

## Ownership Boundaries

| Campaign runtime | `campaignAtlas.js` | Phaser scene, camera, movement, foot collision, interaction handling, debug API |
| Campaign battles | `campaignAtlasEncounters.js` | Encounter selection, challenge content, battle rewards, return callbacks |
| Shared vitals | `combatState.js`, `battleHud.js` | HP, Focus, shared readouts and battle log |
| Sorting Slime arena | `slimeArenaEngine.js`, `slimeArenaPatterns.js`, `slimeArenaConfig.js` | Real-time movement, hazards, shields, phases, animation |

## Campaign Flow

```text
Title / Continue / Chapter Select
  -> enterChapter(number)
  -> startCampaignChapter()
  -> full-screen Phaser atlas
       movement -> precise foot collision
       interaction -> dialogue / puzzle / collectible / exit
       enemy or boss -> campaignAtlasEncounters.js
          -> dedicated arena, queue battle, or Python battle
          -> caller-owned win callback
          -> persist clear flag
          -> resume the same atlas position
```

The battle owns how an enemy is defeated. The atlas owns where the enemy exists and what a win changes in the campaign.

Exploration uses explicit resizing only while its host is visible. Do not change it back to `Phaser.Scale.RESIZE`: a hidden atlas host can collapse to zero while a battle creates another WebGL canvas.

## Collision Contract

The same geometry draws the map and answers collision. A point is walkable when it is inside a region or corridor, outside every blocker, and outside every currently closed gate.

Patchrunner tests four points around his feet and resolves X and Y separately, which allows smooth wall sliding. Foreground arches are independent visual layers and do not change collision.

Every required destination is flood-filled from the spawn with gates treated as eventually open. A second browser audit confirms that blockers reject movement and gate centers become traversable after their requirements are met.

See `docs/CAMPAIGN_ATLAS.md` for the editing workflow.

## Python Runtime

The browser Python surfaces currently translate a deliberately small educational subset in a disposable Web Worker. It supports the loops, indexing, comparisons, assignments, swaps, and returns used by current encounters.

It is not a security sandbox and it is not full Python. Pyodide remains the likely upgrade before problems require dictionaries, sets, recursion, imports, heaps, or broader LeetCode compatibility. See `docs/SAFE_EXECUTION.md`.

## Adding An Encounter

Use a dedicated Phaser arena when movement and enemy behavior are the mechanic. Keep three layers:

1. `*ArenaEngine.js`: deterministic state, movement, collisions, hazards, animation.
2. `*Battle.js`: DOM commands, editor, audio, vitals, and arena callbacks.
3. `campaignAtlasEncounters.js`: campaign reward and return destination.

For a smaller encounter that fits the current queue or Python surface, add a readable config in `campaignAtlasEncounters.js` and a field interaction in `campaignAtlasData.js`.

Never grant campaign progress merely for opening a battle. Persist it only in the normal win callback. Admin completion must use the same callback.

## Verification

- `npm run check` parses every browser JavaScript file.
- `npm run smoke:title` verifies the mural, pixel logo, cursors, options, and pointer-first title.
- `npm run smoke:slime` verifies direct, Arcade, and atlas Slime launch paths plus mechanics and mobile fitting.
- `npm run smoke:workshop` verifies visual editing and local draft persistence.
- `npm run smoke:atlas` verifies all six chapters, collisions, connectivity, puzzles, secrets, animation counts, and responsive canvas framing.
- `npm run verify:web` runs the complete browser build and smoke suite.
- Python unit tests protect the legacy package contracts.

A passing syntax check does not prove a canvas rendered. A screenshot does not prove a route is reachable. Keep both layers.

## Deployment

Vercel follows `vercel.json`:

1. install npm dependencies
2. run `npm run build`
3. publish `web/`

`main` is the production source. A successful Git push starts the Vercel deployment; verify the public URL separately before calling it live.

## Known Technical Debt

- Sorting Slime is still the only fully real-time movement boss.
- Later battle modules duplicate parts of the limited Python translator.
- Workshop exports are not yet consumed directly by the campaign atlas.
- `style.css` remains monolithic.
- Legacy chapter modules and their old tests can be removed after the atlas has had enough production playtesting.
- The Python package name remains misspelled for compatibility.
